# เอกสารข้อกำหนดผลิตภัณฑ์ (PRD)
## โปรเจกต์: ระบบบริหารจัดการสินทรัพย์ถาวรแบบง่าย (Simple Fixed Asset Management System)

### 1. บทนำ (Introduction)
ระบบบริหารจัดการสินทรัพย์ถาวร (Fixed Asset Management System) เป็นเว็บแอปพลิเคชันที่ออกแบบมาเพื่อช่วยให้องค์กรติดตามสินทรัพย์ทางกายภาพ เป้าหมายคือเพื่อทดแทนการติดตามด้วยสเปรดชีตแบบ manual ด้วยระบบศูนย์กลางที่จัดการวงจรชีวิตของสินทรัพย์ ตั้งแต่การจัดหา ไปจนถึงการมอบหมายให้พนักงาน และการรับคืน ระบบนี้เน้นฟังก์ชันการทำงานหลักเพื่อให้ใช้งานง่ายและไม่ซับซ้อน

### 2. วัตถุประสงค์ (Objectives)
- รวบรวมข้อมูลสินทรัพย์ไว้ที่ศูนย์กลาง
- ติดตามผู้ครอบครองสินทรัพย์ปัจจุบัน
- เก็บประวัติการเปลี่ยนแปลงสถานะของสินทรัพย์
- ลดความซับซ้อนในกระบวนการเบิกจ่ายและรับคืนสินทรัพย์

### 3. บทบาทผู้ใช้งาน (User Roles)
- **ผู้ดูแลระบบ/ผู้จัดการสินทรัพย์ (Admin/Asset Manager)**: มีสิทธิ์เข้าถึงเต็มรูปแบบในการจัดการสินทรัพย์ ประเภทสินทรัพย์ และการมอบหมาย
- **ผู้ใช้ (User)**: มีสิทธิ์เข้าถึงฟังก์ชันการดูข้อมูลสินทรัพย์ และการมอบหมายสินทรัพย์

### 4. ข้อกำหนดด้านฟังก์ชัน (Functional Requirements)

#### 4.1 การจัดการประเภทสินทรัพย์ (Asset Type Management)
*ระบบต้องอนุญาตให้จัดหมวดหมู่สินทรัพย์ได้*
- **สร้างประเภทสินทรัพย์**: เพิ่มหมวดหมู่ใหม่ (เช่น แล็ปท็อป, จอภาพ, เฟอร์นิเจอร์, ยานพาหนะ)
- **รายการประเภทสินทรัพย์**: ดูหมวดหมู่ที่มีอยู่ทั้งหมด
- **แก้ไข/ลบประเภทสินทรัพย์**: แก้ไขหรือลบหมวดหมู่ (การตรวจสอบ: ป้องกันการลบหากมีสินทรัพย์ผูกอยู่กับหมวดหมู่นั้น)

#### 4.2 การจัดการทะเบียนสินทรัพย์ (Asset Inventory Management)
*ฐานข้อมูลหลักของรายการสินทรัพย์ทั้งหมด*
- **เพิ่มสินทรัพย์**: ลงทะเบียนสินทรัพย์ใหม่พร้อมรายละเอียด:
    - รหัสสินทรัพย์/Tag ID (ต้องไม่ซ้ำกัน)
    - ชื่อ/รุ่น
    - ประเภทสินทรัพย์ (เลือกจากข้อ 4.1)
    - หมายเลขซีเรียล (Serial Number)
    - วันที่ซื้อ
    - ราคา
    - สถานะปัจจุบัน (ว่าง/พร้อมใช้, กำลังใช้งาน, ส่งซ่อม, ตัดจำหน่าย)
- **การกำหนดรูปแบบรหัสสินทรัพย์**: (TBD)
- **แก้ไขสินทรัพย์**: อัปเดตรายละเอียดสินทรัพย์
- **ลบสินทรัพย์**: ลบรายการสินทรัพย์ (ควรเป็นการลบแบบ Soft delete หรือแค่เปลี่ยนสถานะ)
- **รายการ/ค้นหาสินทรัพย์**: ดูรายการสินทรัพย์พร้อมตัวกรองตามประเภทหรือสถานะ

#### 4.3 การมอบหมายสินทรัพย์ (Asset Assignment / Check-out)
*การติดตามว่าใครถือครองสินทรัพย์อยู่*
- **มอบหมายสินทรัพย์**: เชื่อมโยงสินทรัพย์ที่มีสถานะ "ว่าง/พร้อมใช้" กับพนักงาน/ผู้ใช้
    - ข้อมูลที่ต้องกรอก: ชื่อ/รหัสพนักงาน, วันที่มอบหมาย, หมายเหตุ
- **อัปเดตสถานะ**: เปลี่ยนสถานะสินทรัพย์เป็น "กำลังใช้งาน" (In Use) โดยอัตโนมัติ

#### 4.4 การรับคืนสินทรัพย์ (Asset Return / Check-in)
*กระบวนการรับสินทรัพย์กลับเข้าคลัง*
- **รับคืนสินทรัพย์**: บันทึกการคืนสินทรัพย์จากพนักงาน
    - ข้อมูลที่ต้องกรอก: วันที่คืน, สภาพสินค้า/หมายเหตุ
- **อัปเดตสถานะ**: เลือกสถานะใหม่ด้วยตนเองเมื่อรับคืน (เช่น "ว่าง/พร้อมใช้", "ชำรุด/ส่งซ่อม", "ตัดจำหน่าย")

#### 4.5 แดชบอร์ด (Dashboard) (ทางเลือก/ควรมี)
- **การ์ดสรุปผล**: จำนวนสินทรัพย์ทั้งหมด, สินทรัพย์ที่กำลังใช้งาน, สินทรัพย์ที่ว่างอยู่

### 5. ข้อมูลเอนทิตี (Data Entities - แบบย่อ)
- **AssetType (ประเภทสินทรัพย์)**: `id`, `name` (ชื่อ), `description` (คำอธิบาย)
- **Asset (สินทรัพย์)**: `id`, `code` (รหัส), `name` (ชื่อ), `type_id` (รหัสประเภท), `serial_number`, `status` (สถานะ), `purchase_date` (วันที่ซื้อ), `price` (ราคา)
- **Employee (พนักงาน)**: `id`, `name` (ชื่อ), `department` (แผนก) (อาจเป็นเพียงช่องข้อความธรรมดาในเบื้องต้น หากยังไม่มีระบบพนักงานเต็มรูปแบบ)
- **Transaction/Log (ประวัติรายการ)**: `id`, `asset_id`, `employee_name` (ชื่อผู้ครอบครอง), `action_type` (ประเภทรายการ: เบิกออก, รับคืน), `date` (วันที่), `notes` (หมายเหตุ)

### 6. ข้อกำหนดที่ไม่ใช่ฟังก์ชัน (Non-Functional Requirements)
- **ความง่ายในการใช้งาน (Usability)**: อินเทอร์เฟซสะอาดตา ใช้งานง่าย (ใช้ Material UI หรือ Tailwind)
- **ประสิทธิภาพ (Performance)**: โหลดรายการสินทรัพย์ได้รวดเร็ว
- **ความน่าเชื่อถือ (Reliability)**: การติดตามสถานะต้องแม่นยำ (สินทรัพย์ต้องไม่สามารถถูกมอบหมายซ้ำได้หากสถานะคือ "กำลังใช้งาน")
